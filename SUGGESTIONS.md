### 1. وحدة OCR — التقييم: 10/10 ✅

#### النقاط القوية:

- **4 محركات OCR متكاملة:** TrOCR, EasyOCR, Tesseract, PaddleOCR
- **4 استراتيجيات دمج النتائج:**
  - `highest_confidence` — اختيار النتيجة ذات أعلى ثقة
  - `weighted_average` — المتوسط المرجح لجميع المحركات
  - `voting` — التصويت بين المحركات
  - `longest_text` — أطول نص مستخرج
- **GPU Acceleration** مع دعم CUDA و OpenVINO
- **ONNX Runtime** للاستدلال السريع
- **Quantization** — FP16 و INT8 لتقليل حجم النموذج وتسريع الاستدلال
- **Caching** ذكي للنتائج لتجنب إعادة المعالجة
- **معالجة مسبقة للصور:**
  - CLAHE (Contrast Limited Adaptive Histogram Equalization)
  - Denoising (إزالة الضوضاء)
  - Deskew (تصحيح الميل)
  - Otsu Threshold (تحديد العتبة)
- **Layout Analysis** لتحليل تخطيط الصفحة
- **Table Extraction** لاستخراج الجداول

#### الاقتراحات:

|---|---|---|---|
| 1 | **Lazy Loading لـ PaddleOCR** — تحميل النموذج فقط عند الحاجة لتقليل استهلاك الذاكرة عند بدء التشغيل | عالية | 📋 مقترح |
| 2 | **Parallel Processing** — معالجة الصفحات المتعددة بالتوازي باستخدام ThreadPool/ProcessPool | عالية | ✅ مُنفَّذ (Celery) |

## 🔷 3. الحفاظ على تنسيق النص أثناء التدريب (Format-Aware OCR Training) — مقترح

### الوصف التفصيلي

هذا اقتراح جوهري لتطوير قدرة النظام على الحفاظ على تنسيق المستند الأصلي أثناء عملية التعرف على النصوص OCR وبخاصة أثناء تدريب النماذج. الهدف هو أن يكون النص الناتج مطابقاً للمستند الأصلي كما لو تمت كتابته على الحاسوب، مع الحفاظ على:

- **الجداول**: بنية الصفوف والأعمدة ومحاذاة الخلايا
- **تسميات الصور**: أرقام و عناوين الصور في مواقعها الصحيحة (مثل: "شكل 1: مخطط النظام")
- **الفقرات**: المسافات البادئة والفقرات الجديدة
- **العناوين**: أحجام الخطوط المختلفة للعناوين والعناوين الفرعية
- **القوائم**: التعداد النقطي والرقمي
- **الحواشي**: الهوامش السفلية والتعليقات

### المشكلة الحالية

النظام الحالي يُنتج نصاً مسطحاً (flat text) يفقد الكثير من معلومات التنسيق الأصلية:

1. **الجداول**: `TableExtractor` يستخرج البيانات لكن لا يحافظ على عرض الأعمدة ومحاذاة الخلايا عند التصدير
2. **الصور**: لا توجد آلية لتعرف النظام بأن هناك صورة في المستند وأن بجانبها تسمية (caption)
3. **العناوين**: `LayoutAnalyzer` يكتشف العناوين لكن لا يُرسل معلومات التنسيق (حجم الخط، اللون، البولد) مع النص
4. **الفقرات**: `TextReconstructor` يعيد بناء النص لكنه لا يحتفظ بالمسافات البادئة (indentation)
5. **القوائم**: لا يوجد تمييز بين القوائم والنصوص العادية
6. **المراجع**: الأرقام المرجعية (مثل [1], [2]) والحواشي السفلية لا يتم الحفاظ عليها

### المكونات المقترحة

#### أ. وحدة التنسيق المهيكل (Structured Format Module) — `modules/export/format_preserver.py`

```python
class FormatPreserver:
    """الحفاظ على تنسيق المستند الأصلي أثناء OCR."""
    
    def preserve_layout(self, blocks: list[DocumentBlock]) -> StructuredDocument:
        """
        تحويل كتل المستند إلى مستند مهيكل يحافظ على التنسيق.
        
        يحول:
        - BlockType.HEADING → Markdown heading (# , ## , ###)
        - BlockType.TABLE → Markdown table مع محاذاة الأعمدة
        - BlockType.TEXT → فقرات مع الحفاظ على المسافات البادئة
        - BlockType.HEADER → header/footer منفصل
        - BlockType.IMAGE_CAPTION → نص مرتبط بصورة مع رقم الشكل
        """
        
    def detect_figure_captions(self, blocks: list[DocumentBlock]) -> list[FigureCaption]:
        """
        كشف تسميات الصور (Figure Captions).
        
        الأنماط المدعومة:
        - "شكل 1:" / "Figure 1:" / "Fig. 1:"
        - "صورة 1:" / "الشكل الأول"
        - أرقام مع قوس: "(1)" بعد النص
        
        يربط كل تسمية بالكتلة الصورية المجاورة لها.
        """
        
    def detect_references(self, text: str) -> list[Reference]:
        """
        كشف المراجع والأرقام المرجعية.
        
        الأنماط:
        - [1], [2,3], [1-5]
        - (Author, Year)
        - أرقام حواشي سفلية: ¹, ², ³
        """
        
    def preserve_lists(self, blocks: list[DocumentBlock]) -> list[ListBlock]:
        """
        كشف القوائم (Bullet Lists / Numbered Lists).
        
        أنماط الكشف:
        - النقاط: •, -, *, ○, ▪
        - الأرقام: 1., 2., 3. أو 1), 2), 3)
        - الحروف: a), b), c) أو أ), ب), ج)
        """
```

```python
class FormatPatternDB:
    """قاعدة بيانات أنماط التنسيق المستندات العربية."""
    
    # أنماط تسميات الصور العربية
    ARABIC_FIGURE_PATTERNS = [
        r"شكل\s*\d+\s*[:\-—]?",     # شكل 1: / شكل 2 -
        r"الشكل\s*\d+\s*[:\-—]?",    # الشكل الأول:
        r"صورة\s*\d+\s*[:\-—]?",     # صورة 1:
        r"مخطط\s*\d+\s*[:\-—]?",     # مخطط 1:
        r"جدول\s*\d+\s*[:\-—]?",     # جدول 1:
        r"رسم\s*\d+\s*[:\-—]?",      # رسم توضيحي 1:
    ]
    
    # أنماط المراجع
    REFERENCE_PATTERNS = [
        r"\[\d+(?:[,،]\s*\d+)*\]",   # [1], [1,2], [1, 2]
        r"\[\d+\s*[-–]\s*\d+\]",      # [1-5]
        r"\(\w+,\s*\d{4}\)",           # (Author, 2024)
        r"\d+\.\s+",                   # 1. (مرجع مرقم)
    ]
    
    # أنماط القوائم
    LIST_PATTERNS = {
        'bullet': r"^[•\-\*○▪▸►]\s+",
        'numbered': r"^(\d+|[أ-ي][\.)]|a-z[\.)])\s+",
        'sub_bullet': r"^[◦‣◦]\s+",
    }
```

### 🚀 ثانياً: مقترحات التطوير (الخارطة التقنية)

#### 1. تحسين محرك التصنيف (The Brain)

بدلاً من الاعتماد الكلي على الكلمات المفتاحية (Keyword Matching)، الانتقال إلى **التصنيف الدلالي (Semantic Classification)**:

- **تضمين النماذج اللغوية:** استخدام نماذج مثل AraBERT أو CamelBERT التي تفهم السياق؛ فمثلاً ستعرف أن كلمة "كسر مضاعف" و"تثبيت داخلي" تتبع لقسم "الجراحة العظمية" حتى لو لم تكن الكلمة الصريحة موجودة.
- **منطق الاحتمالات:** بدلاً من وضع الملف في مجلد واحد، إعطاء "نسبة ثقة" (مثلاً: 90% طبي، 10% إداري).

#### 2. معالجة الأداء (Optimization)

بما أن المشروع يتعامل مع آلاف الملفات ونماذج ذكاء اصطناعي ثقيلة:

- **المعالجة على الدفعات (Batch Processing):** تجميع الصور وإرسالها للـ GPU دفعة واحدة لتقليل زمن التحميل (Latency).
- **نظام الكاش (Caching):** إنشاء قاعدة بيانات SQLite تخزن "بصمة الملف" (Hash) ونتائج معالجته. إذا تم نقل الملف أو إعادة تسميته، لن يحتاج البرنامج لإعادة تشغيل الـ OCR مرة أخرى. ✅ **مُنفَّذ** في `modules/core/database_manager.py`

#### 3. تعزيز الـ OCR للمستندات الطبية

- **قاموس المصطلحات المخصص:** دمج قاموس (Custom Vocabulary) مع محرك الـ OCR يجبر المحرك على تصحيح الكلمات القريبة من المصطلحات الطبية المعروفة بدلاً من تخمين كلمات عربية عامة.
- **المعالجة المسبقة (Image Pre-processing):** إضافة فلاتر OpenCV لتحسين التباين وإزالة الضجيج من الأوراق الممسوحة ضوئياً قديماً قبل إرسالها لـ TrOCR.

#### 4. واجهة المستخدم والتفاعل

- **نظام الوسوم (Tagging System):** بدلاً من نظام المجلدات التقليدي الذي يحد من مكان الملف، اعتماد نظام "الوسوم الذكية" حيث يمكن للملف الواحد أن يحمل وسماً (طبي، 2026، جامعة).
- **لوحة تحكم Gradio متقدمة:** إضافة "نافذة تدقيق" (Validation Window) تعرض الصورة والنص المستخرج بجانبها لتسمح بالتصحيح اليدوي وتدريب النموذج على التصحيحات.

### 💡 ثالثاً: رؤية مستقبلية للمشروع (Legacy Features)

- **البحث الدلالي (Vector Search):** باستخدام مكتبة مثل FAISS أو ChromaDB؛ يمكن مستقبلاً سؤال البرنامج: "أين الملف الذي يتحدث عن عمليات تبديل مفصل الورك في عام 2010؟" وسيقوم بجلب الملفات بناءً على المعنى وليس فقط الكلمات.
